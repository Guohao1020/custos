# Custos · Agent 原生密钥与权限引擎 · 需求设计稿（PRD v2）

> **项目代号**：Custos（拉丁语「守护者」）
> **一句话定位**：为 **Nacos / Spring Cloud 生态**打造的、**自托管**的 **AI Agent 身份 · 密钥 · 权限**统一引擎——借鉴 Vault/OpenBao，但从 Agent 场景重新设计，**自研密钥引擎**，**Nacos 作注册/控制面**。

| 项 | 内容 |
|----|------|
| 项目代号 | **Custos** |
| 版本 | v2.0（方向调整后草案） |
| 状态 | 评审中 |
| 引擎路线 | **完全自研**（不依赖 Vault/OpenBao 内核） |
| 控制面 | **Nacos 3.x** |
| 适用栈 | Java / Spring Cloud · Kubernetes |
| 校订日期 | 2026-06 |

> 说明：本 PRD 取代此前《企业内部 AI 平台 PRD（治理平台）》中关于密钥/权限的部分。项目方向已从「AI 治理大平台」收窄聚焦为「**Agent 密钥与权限引擎**」。原治理平台文档（注册中心/网关/交付灰度）作为更广背景资料保留，但本项目首期只做本 PRD 范围。

## 1. 背景与定位

AI Agent（Claude、Codex、自治 Agent）要操作企业内部系统，就必须获得数据库账号、内部系统账号、AK/SK 等敏感凭证与权限。Agent 与传统应用/人有本质差异：**身份临时且海量、代表用户行动（委托）、权限需绑定到工具/动作、行为不确定需即时授权、密钥绝不能进入 LLM 上下文**。现有密钥系统（Vault/OpenBao）是为应用/人设计的，Agent 场景适配生硬。

### 1.1 市场现状（已调研，必须正视）

同方向已有玩家：**Infisical Agent Vault**（开源，研究预览，"agents never see the secret"）、**Aembit**（商业 Agent 身份经纪）、**SPIFFE/SPIRE**（开源工作负载身份标准）、**Cerbos**（Agent/MCP 权限策略）。标准也在收敛：MCP **SEP-835**（工具级 scope）、**NIST AI Agent 标准**、**CoSAI Workstream 4**（Agentic IAM）。

> 因此 Custos **不以「第一个 Agent 版 Vault」自居**，差异化钉死在一处别人没有的地方 ↓

### 1.2 差异化护城河：Nacos-native

上述竞品无一建在 Nacos 上。而国内大量 Java/Spring Cloud 企业的注册配置中心正是 Nacos。Custos 把 Agent 身份/密钥/权限做成 **Nacos 原生**——对这批企业是「装上即用」，对手是「另接一套」。

**定位语**：*Custos —— 为 Nacos / Spring Cloud 生态打造的、自托管的 Agent 身份·密钥·权限统一引擎。*

## 2. 目标与非目标

### 2.1 目标

- **G1**：自研一个**安全的密钥引擎**（加密存储、seal/unseal、动态凭证、租约/撤销、防篡改审计）。
- **G2**：**Agent 原生身份**：每 Agent / 每会话临时身份；**OBO 委托**（权限 = 用户 ∩ Agent，取最小）。
- **G3**：**权限/系统权限管理**：RBAC + ABAC/PBAC，工具/动作级 scope（对齐 MCP SEP-835），高危 JIT + 人工审批。
- **G4**：**Nacos 作控制面**：注册 Agent/资源/策略，策略热更新 = **秒级权限变更与吊销**，namespace 隔离。
- **G5**：**密钥不进 LLM 上下文**：MCP-native 的 secretless 经纪，凭证带 TTL、可轮换。
- **G6**：面向 Java/Spring Cloud + K8s 的开箱即用 SDK 与部署。

### 2.2 非目标（本期不做）

- 不做 AI 网关、注册中心本体、RAG、Agent 编排（与本项目互补，非本项目）。
- 不做通用人类 SSO/IDP（聚焦非人类/Agent 身份；人通过既有 SSO 接入做委托源）。
- 不自创密码学算法（用经过审计的密码库实现标准算法）。

## 3. 用户角色与场景

| 角色 | 诉求 | 场景 |
|------|------|------|
| AI Agent（Claude/Codex/自治） | 代表用户安全访问内部资源 | 查只读库、调内部系统、用云资源 |
| 研发工程师 | 用 Agent 提效且不泄密 | 自然语言查数据/排障，背后短时凭证 |
| 平台/安全管理员 | 统一治理 Agent 访问 | 定策略、分级、审批、秒级吊销、审计 |
| 应用/服务 | 程序化获取凭证 | Spring 服务经 SDK 取动态凭证 |

## 4. 总体架构

```
            ┌──────────── 消费者 ────────────┐
            │  Claude · Codex · 自治 Agent · Spring 服务 │
            └───────────────┬───────────────┘
                  MCP / SDK  │ (secretless：拿结果，不拿密钥)
        ┌───────────────────▼────────────────────┐
        │            Custos 引擎（自研）            │
        │  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
        │  │ 身份层    │ │ 策略层PDP │ │ 经纪层PEP │ │
        │  │ Identity │ │  AuthZ   │ │ Secrets  │ │
        │  └──────────┘ └──────────┘ └──────────┘ │
        │  ┌───────────────────────────────────┐  │
        │  │ 引擎内核：Barrier 加密 · Seal/Unseal │  │
        │  │ 存储 · 租约/撤销 · 防篡改审计        │  │
        │  └───────────────────────────────────┘  │
        └───────┬───────────────────────┬─────────┘
       注册/策略热更新                  签发/代理
        ┌───────▼────────┐      ┌───────▼──────────┐
        │ Nacos 3.x 控制面 │      │ 受保护资源        │
        │ Agent/资源/策略  │      │ DB · 内部系统·AK/SK│
        │ 注册 + 热更新吊销 │      └──────────────────┘
        └─────────────────┘
基础设施： Kubernetes · 存储后端(MySQL/嵌入式) · ELK/OTel
```

经典 **PDP/PEP** 模型：Nacos = 控制面（注册 + 策略分发 + 热更新）；策略层 = PDP（判 Agent ∩ 用户 ∩ 资源 ∩ 风险）；经纪层 = PEP（签发短时凭证或 secretless 转发）。

## 5. 功能需求（按模块）

### FR-ENGINE · 自研引擎内核（P0，安全核心）

| 编号 | 需求 |
|------|------|
| E1 | **Barrier 加密**：所有数据落盘前用 AES-256-GCM（经审计密码库）加密 |
| E2 | **密钥层级**：master key → barrier key；master key 绝不明文落盘 |
| E3 | **Seal/Unseal**：Shamir 分片解封 + 可选 KMS 自动解封；启动默认 sealed |
| E4 | **可插拔存储后端**：首版 MySQL（企业已有）；数据全密文 |
| E5 | **租约管理**：TTL 跟踪、到期自动撤销、续约 |
| E6 | **动态凭证签发**：连后端现场建临时账号、到期销毁 |
| E7 | **防篡改审计**：哈希链 / 只追加，日志不可被悄悄改 |
| E8 | **内存安全**：密钥用完清零、禁 swap、明文存活最小化 |

### FR-ID · Agent 身份（P0/P1）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| ID1 | 每 Agent / 每会话临时身份签发（JWT / SPIFFE 风格 SVID） | P0 |
| ID2 | 认证方法：OIDC/JWT、K8s SA、（后续）SPIFFE | P0 |
| ID3 | **OBO 委托**：用户令牌 + Agent 身份 → 派生作用域令牌 = 两者权限交集 | P1 |
| ID4 | 身份在 Nacos 注册与查询 | P0 |

### FR-SEC · 密钥经纪（P0/P1）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| S1 | 动态 DB 只读凭证：CREATE/DROP + TTL（默认 1h/最大 4h） | P0 |
| S2 | **secretless 经纪**：MCP 服务执行并只回结果，密钥不返回 LLM | P0 |
| S3 | KV 密钥、AK/SK 签发与**定期轮换** | P1 |
| S4 | Spring 服务经 SDK 直取动态凭证、随租约续期/失效 | P1 |

### FR-AUTHZ · 权限/系统权限（P0/P1）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| A1 | 策略模型：RBAC + ABAC/PBAC（按意图/上下文/风险判定） | P0 |
| A2 | **工具/动作级 scope**，对齐 MCP SEP-835 | P0 |
| A3 | 资源分级 + Agent 自治等级；高危 **JIT + 人工审批** | P1 |
| A4 | PDP/PEP 分离；策略评估可解释（给出准/拒原因） | P0 |

### FR-NACOS · 控制面集成（P0，差异化核心）

| 编号 | 需求 |
|------|------|
| N1 | 注册：Agent 身份、受保护资源目录、权限策略、引擎/经纪实例 |
| N2 | **策略以 Nacos 配置下发，热更新 = 秒级权限变更与吊销** |
| N3 | namespace 隔离（团队/环境）；group 分组 |
| N4 | 经纪/引擎组件经 Nacos 服务发现 |

### FR-IF · 接口与 SDK（P0/P1）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| IF1 | **MCP-native**：以 MCP server/工具暴露，Claude/Codex 标准接入 | P0 |
| IF2 | Spring Boot Starter / SDK（注解取凭证、身份注入） | P1 |
| IF3 | `custos` CLI（管理策略、资源、解封、审计查询） | P1 |

### FR-AUDIT · 审计（P0）

| 编号 | 需求 |
|------|------|
| AU1 | 每次「决策 + 访问」留痕：Agent + 用户 + 任务 + 资源 + 结果 |
| AU2 | 防篡改（见 E7）；导出 ELK/OTel；异常/超额告警 |

## 6. 非功能需求（NFR）

| 类别 | 要求 |
|------|------|
| **安全（重中之重）** | 不自创密码学（用审计库实现标准算法）；Barrier 加密；master key 经 Shamir/KMS 保护、不明文落盘；防篡改审计；内存清零禁 swap；**上生产前做外部安全审计**；威胁建模 |
| **吊销正确性** | 租约可靠跟踪 + 传播；Nacos 策略变更秒级生效，可验证 |
| **可用性** | 首版单节点；后续 Raft HA + 强一致（租约不丢不重）；解封流程明确 |
| **性能** | 凭证签发与策略评估低延迟；secretless 转发开销可控（压测 P95） |
| **可观测** | 指标 Prometheus；审计/链路 ELK/OTel；解封/密封状态可监控 |
| **兼容** | Nacos 3.x；Spring Boot↔Cloud↔Cloud Alibaba 版本对齐；K8s 部署 |
| **合规** | 凭证最小只读权限；密钥不进 LLM/不进代码仓库；可追溯到用户 |

## 7. MVP（v0.1）· 一条纵向线

> **目标**：用最薄的一条端到端链路，证明「身份 + 权限 + 密钥 + Nacos 秒级吊销」四件事，且密钥不进 LLM。

**演示链路**：
1. Claude/Codex 代表某用户，请求查询一个只读数据库。
2. Custos 签发该 Agent 的 per-task 身份（JWT）。
3. 策略层（策略存 Nacos）校验 Agent ∩ 用户 ∩ 资源 ∩ 风险 → 准/拒并给原因。
4. 经纪层用自研引擎现场签发 1h 只读 DB 凭证，**secretless 执行查询、只回结果**。
5. 全程审计留痕（Agent+用户+任务+SQL 摘要）。
6. 在 Nacos 改一下策略 → 该 Agent 访问**秒级被吊销**（可演示的对 Vault 的优势）。

**v0.1 引擎最小集**：Barrier 加密 + Shamir 解封 + MySQL 存储 + 动态 DB engine + 租约/撤销 + 哈希链审计；认证用 JWT；策略用基础 RBAC（存 Nacos）；MCP-native 经纪服务。

| 版本 | 重点 | 验收 |
|------|------|------|

## 9. 风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| **自研密钥引擎安全风险** | 🔴 极高 | 用审计密码库、不造算法；威胁建模；v0.4 前外部安全审计；seal/unseal 与审计完整性重点投入 |
| master key 泄漏 = 全盘沦陷 | 🔴 | Shamir 分片/KMS；不明文落盘；最小化明文存活 |
| 吊销/租约不可靠 | 🟠 | 重点测试；HA 用强一致；秒级吊销做端到端验证 |
| 同方向竞品（Infisical Agent Vault 等） | 🟠 | 钉死 Nacos-native + 自托管 + 身份/密钥/权限一体差异化 |
| 标准仍在演进（MCP SEP-835/NIST/CoSAI） | 🟡 | 对齐标准、保持可适配；接口抽象 |
| 自研工作量大、周期长 | 🟠 | 严格 MVP 纵向线；先单节点；引擎能力按版本逐步补 |

## 10. 命名 · 许可 · 仓库

- **名称**：Custos；CLI `custos`；包名 `io.custos.*`。
- **许可证**：建议 Apache-2.0（企业友好）。注意：持有密钥的系统，发布前明确安全责任声明与披露策略。
- **仓库结构（建议）**：`engine/`（自研内核）· `identity/` · `authz/`（PDP）· `broker/`（PEP/MCP）· `nacos/`（控制面集成）· `sdk/`（Spring Boot Starter）· `cli/` · `examples/`（compose/helm + demo 纵向线）。

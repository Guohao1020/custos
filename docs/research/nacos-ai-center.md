## Nacos 3.2 AI Admin Center - MCP Registry & Resource Import

### MCP Server Registry & CRUD  · `high` · 落点：v0.6 (post v0.5 resource-intake phase): Register Custos as MCP server in Nacos; emit tool specs carrying ABAC/permission metadata.
- **干啥**：Nacos native registry for MCP (Model Context Protocol) servers. Manages server metadata (name, protocol, description), tool specifications (with input/output schemas, annotations), resource specifications (plain or encrypted), endpoint configuration (direct address:port or service reference), and versioning. Supports multiple protocols: stdio, SSE, streamable HTTP, HTTP, Dubbo.
- **API**：GET /v3/admin/ai/mcp/list?pageNo={p}&pageSize={s}&search={q} | GET /v3/admin/ai/mcp?mcpId={id}&mcpName={n}&version={v} | POST /v3/admin/ai/mcp (create) | PUT /v3/admin/ai/mcp (update) | DELETE /v3/admin/ai/mcp?mcpId={id}&mcpName={n}&version={v}. All require @Secured(action=READ/WRITE, signType=AI, apiType=ADMIN_API).
- **数据模型**：McpServerBasicInfo (id, name, protocol, description, repository, packages, icons, websiteUrl, versionDetail, remoteServerConfig, localServerConfig, enabled, status, capabilities); McpToolSpecification (specificationType, encryptData, tools[], toolsMeta{}, securitySchemes[]); McpResourceSpecification (specificationType, encryptData, resources[], resourceTemplates[], extensions); McpEndpointSpec (type=DIRECT|REF, data map with address/port or namespaceId/groupName/serviceName).
- **Custos 怎么用**：Custos registers Custos MCP server (executing query_db + policy_check + audit_log tools) into Nacos registry. Each tool metadata embeds Custos permission/ABAC requirements (e.g., tool 'query_db' requires 'datasource:read' + risk scoring). At v0.5-v0.6, Agents discover Custos in Nacos, fetch tool specs with embedded policy markers, then negotiate credentials (AK·SK or JWT) before invoking Custos tools. Enables: (1) tool discoverability, (2) permission envelope on each tool invocation, (3) centralized governance of AI tool capabilities.

### AI Resource Import Framework (unified interface for external sources)  · `high` · 落点：v0.5 (resource intake + multi-engine): Build 'custos-resource-importer' plugin; register with Nacos AI import framework. Later: v0.7 extend to support multi-Custos cluster sources.
- **干啥**：Three-stage bulk import pipeline for AI resources (MCP servers, skills, etc.) from external sources (official MCP registry, skills.sh, custom plugins). Stages: (1) Search external source (cursor pagination), (2) Validate selected candidates (dry-run, conflict check, returns validationToken), (3) Execute import (apply artifact, overwrite option, skip-invalid flag).
- **API**：GET /v3/admin/ai/import/sources?resourceType={mcp|skill} -> List[AiResourceImportSourceInfo]; POST /v3/admin/ai/import/search (AiResourceImportSearchRequest) -> AiResourceImportSearchResponse; POST /v3/admin/ai/import/validate (AiResourceImportValidateRequest) -> AiResourceImportValidateResponse (with validationToken); POST /v3/admin/ai/import/execute (AiResourceImportExecuteRequest + validationToken) -> AiResourceImportExecuteResponse. All WRITE ops @Secured(action=WRITE, signType=AI).
- **数据模型**：AiResourceImportSourceInfo (sourceId, displayName, description, pluginName, resourceTypes[], enabled, capabilities[]); AiResourceImportCandidateItem (externalId, name, version, description, metadata{}); AiResourceImportItem (externalId, name, version, metadata{}); AiResourceImportValidateRequest (namespaceId, resourceType, sourceId, selectedItems[], overwriteExisting, options{}); AiResourceImportExecuteRequest (idem + skipInvalid, validationToken); AiResourceImportValidateResponse (validationToken, items[AiResourceImportValidationItem]); AiResourceImportExecuteResponse (items[AiResourceImportResultItem with status enum)).
- **Custos 怎么用**：Custos integrates a custom import source plugin (e.g., 'custos-secrets-registry') that scans Custos-managed secrets/credentials across multiple tenants or environments. Admin discovers 'custos-secrets-registry' in /sources, searches candidates (e.g., all AK·SK pairs from prod DB), validates import (check duplication, permission mapping), then bulk-imports into Nacos as governed resources. Each imported secret/credential artifact carries tenant+RBAC labels. Decouples resource ingestion from Nacos: operators bulk-ingest Custos-brokered resources into Nacos for centralized discovery by Agents.

### MCP Server Tool Metadata Enrichment (specification type control)  · `medium` · 落点：v0.6 (tool metadata governance): Ship Custos with 'normal' type, tool.annotations carrying policy URIs (e.g., /custos/api/policy?toolId=query_db). Post-v1.0 consider encrypted envelopes for high-security deployments.
- **干啥**：Tool specification supports 'specificationType' enum (normal/encrypted) and optional 'encryptData' blob. When type='encrypted' (or vendor like 'encrypt-kms'), Nacos persists encryptData as-is and does NOT parse/validate tools, securitySchemes inside; client must handle decryption. Enables tool specs to carry Custos-encrypted permission envelopes or policy attachments without Nacos parsing.
- **API**：Embedded in McpToolSpecification { specificationType?: string (default 'normal'), encryptData?: EncryptObject{algorithm, payload, metadata}, tools[]?, toolsMeta{}?, securitySchemes[]? }. Applies to POST /v3/admin/ai/mcp and PUT /v3/admin/ai/mcp bodies.
- **数据模型**：EncryptObject (algorithm: string, payload: string/bytes, metadata{}); McpToolSpecification fields: specificationType, encryptData, tools (only parsed if type='normal'); legacy: tools can embed _meta field per MCP spec for custom metadata.
- **Custos 怎么用**：When registering Custos MCP server, set tool spec specificationType='encrypted' and encryptData={algorithm:'custos-hmac-sha256', payload:<encrypted policy + permission matrix>}. Nacos stores as-is. Client-side: Agent calls Nacos /mcp/get, receives tool spec with encryptData, passes encrypted envelope to Custos TokenService for validation before tool dispatch. Alternate: set specificationType='normal' and embed policy in tool.annotations (non-encrypted, lower friction, but visibility tradeoff).

### MCP Endpoint Multi-Mode Discovery (direct vs service reference)  · `high` · 落点：v0.6 (deployment patterns): Implement DIRECT for single-Nacos deployments, support REF for multi-cluster scenarios.
- **干啥**：MCP server endpoint can be specified in two modes: (1) DIRECT: Nacos creates a Naming service and registers endpoint instances with address:port metadata, OR (2) REF: MCP resource references an existing pre-registered Naming service (namespaceId, groupName, serviceName). Enables decoupling MCP metadata from endpoint infrastructure.
- **API**：McpEndpointSpec { type: 'DIRECT' | 'REF', data: {address, port} | {namespaceId, groupName, serviceName} }. Set in POST /v3/admin/ai/mcp and PUT /v3/admin/ai/mcp request bodies.
- **数据模型**：McpEndpointSpec type enum: MCP_ENDPOINT_TYPE_DIRECT, MCP_ENDPOINT_TYPE_REF. Constants: MCP_SERVER_ENDPOINT_GROUP='mcp-endpoints', MCP_SERVER_ENDPOINT_CLUSTER='DEFAULT_CLUSTER', MCP_SERVER_ENDPOINT_ADDRESS='address', MCP_SERVER_ENDPOINT_PORT='port'.
- **Custos 怎么用**：At v0.6, Custos MCP host (custos-host) exposes 3 endpoints (policy, audit, query_db). Operator chooses: (1) DIRECT mode: Custos-host auto-registers itself in Nacos under group 'mcp-endpoints', service 'custos-mcp-server', instances {address: <host-ip>, port: <grpc-port>}; OR (2) REF mode: operator pre-registers Custos host in Nacos Naming, then MCP server points to that service. REF allows Custos to be shared across multiple Nacos clusters or managed outside Nacos.

### Versioning & Release Lifecycle (version resolution for MCP servers)  · `high` · 落点：v0.6+: Manage Custos MCP server version releases; emit ACTIVE/DEPRECATED status.
- **干啥**：MCP servers support multiple versions stored independently. Public APIs query by mcpName without version -> resolves latest published version. Admin can query specific versions. Version detail includes release date, stability status (ACTIVE|DEPRECATED), compatible clients, breaking changes.
- **API**：GET /v3/admin/ai/mcp?mcpName={name} -> latest version; GET /v3/admin/ai/mcp?mcpName={name}&version={v} -> specific version. Response: McpServerDetailInfo includes ServerVersionDetail {releaseDate, status, compatibleClients, breakingChanges, ...}.
- **数据模型**：ServerVersionDetail (versionNumber, releaseDate (format: 'yyyy-MM-dd\'T\'HH:mm:ss\'Z\''), status: 'ACTIVE'|'DEPRECATED', compatibleClients[], breakingChanges[]). Stored: MCP_SERVER_VERSION_DATA_ID_SUFFIX='-mcp-versions.json' per server.
- **Custos 怎么用**：Custos releases MCP server as versions (v0.5, v0.6-rc1, v1.0). Each version can signal policy/ABAC schema changes (breaking). Agents query latest Custos MCP, get version info, decide compat before invoking. Enables: (1) graceful rollout of new policy models, (2) deprecation warnings, (3) client-side version negotiation.

### Publish Pipeline (资源发布审核编排)  · `medium` · 落点：v0.6
- **干啥**：Nacos 的通用发布审核管道引擎，用于 Prompt/Skill/AgentSpec 等 AI 资源在发布前的审核拦截。架构：SPI 链式插件 + 数据库持久化执行记录 + 异步回调通知。流程：submit() 触发 → PublishPipelineContext 生成 → PublishPipelineExecutor 按 preferOrder 顺序同步执行各节点 → 每节点返回 passed/rejected + message + checkpoints → 一旦某节点 failed 停止链 → 最终 APPROVED/REJECTED 状态回调给资源服务，资源服务根据结果 publish 或 rollback (rejected 时版本回滚为 draft)。
- **API**：Admin-only Query: GET /v3/admin/ai/pipeline/list?resourceType=PROMPT&resourceName=&version=&pageNo=&pageSize= (列表) | GET /v3/admin/ai/pipeline/detail?pipelineId=<executionId> (查看单次执行详情). Pipeline 节点配置无 HTTP 端点，而是通过 SPI properties 配置在 pipeline.yaml 全局配置文件。
- **数据模型**：PublishPipelineContext: {resourceType (PROMPT/SKILL/AGENTSPEC), resourceName, namespaceId, version, files: [{filePath, content}]}. PublishPipelineResult: {passed: bool, message: String, type (TEXT/MARKDOWN), checkpoints: [{name, status, detail}]}. PipelineExecution (DB): {executionId, resourceType, resourceName, namespaceId, version, status (IN_PROGRESS/APPROVED/REJECTED), pipeline: [PipelineNodeResult], createTime, updateTime}.
- **Custos 怎么用**：Custos 的价值【中-可选】。理由：(1) Pipeline 是一个通用的异步审核编排框架，Custos 的权限决策（三态：ALLOW/REQUIRE_APPROVAL/DENY）可以实现为 Pipeline 的一个 stage node。(2) 用法示例：Custos 暴露 CustomApprovalPipelineService implements PublishPipelineService，在 execute(context) 中调用 Custos PDP 做决策，若 REQUIRE_APPROVAL 则 reject("需要人工审批")，若 ALLOW 则 pass("策略通过")。(3) 这样 Nacos Prompt 发布时自动纳入 Custos 权限控制，无需修改 Prompt 服务本身。(4) 限制：Pipeline 当前是发布前审核，不支持事后审计通知回调（Custos 的 AuditLog 写入由 Custos 内部完成）。(5) 不应该让 Custos 管理 Pipeline 配置本身（nodes/order），这应该留给 Nacos 平台管理。

### Custos as Native PEP/Audit Layer in Nacos AI Stack  · `high` · 落点：v0.5/v0.6 (end-to-end + observable) - v0.5 query_db tool operational; v0.6 Custos OperatorService pushes policies to Nacos; audit logs streamed to Nacos
- **干啥**：Positioning: Custos is the 'identity + secrets + authz + audit' enforcement point that ALL agent↔resource interactions flow through when using Nacos AI ecosystem. Nacos handles discovery (who/what exists); Custos handles permission + credential issuance (who can access what + with what secret).
- **API**：Custos query_db (secretless) + policy evaluation endpoint; MCP query_db tool; integration with Nacos OperatorService/AdminAPI for policy push; audit event stream to Nacos config/naming audit sinks
- **Custos 怎么用**：Every agent tool invocation via Nacos MCP Registry first flows through Custos PEP (Policy Enforcement Point): (1) Agent identity authenticated via Custos token; (2) Custos ABAC/approval policy evaluates action (read-only, approve-required, deny); (3) If approved, Custos issues time-bound DB/API credential; (4) Audit logged. Nacos becomes credential-agnostic; Custos becomes policy/audit-aware.
